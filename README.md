# PirateSwap closed bot — деплой и эксплуатация

Закрытый Telegram-бот для фиксированного списка из 3 человек. Мониторит появление
кейсов на PirateSwap и шлёт алерты. Мониторинг и рассылка разделены: опрос сайта
идёт всегда, `/stop` отключает уведомления только нажавшему.

## Состав репозитория

| Файл | Назначение |
|------|-----------|
| `pirateswap_bot.py` | весь бот (поллер + Telegram + HTTP /health) |
| `requirements.txt` | пустой (stdlib-only), нужен Render для распознавания Python |
| `render.yaml` | Blueprint для Render (опционально) |
| `.gitignore` | чтобы не закоммитить state/логи/`.env` |

`notify_state.json` и `events.jsonl` создаются в рантайме — **не коммить их**.

---

## ⚠️ Шаг 0. Отозвать засвеченный токен

Старый токен был захардкожен в исходнике и засветился. Открой **@BotFather →
`/revoke`** для этого бота, получи **новый** токен. В коде токена больше нет —
он читается только из переменной окружения.

---

## Шаг 1. Активация чатов (один раз)

Telegram не даёт боту писать первым. Поэтому **каждый из троих** должен один раз
открыть бота и нажать **/start**. Это не «регистрация новых людей» — белый список
жёстко зашит в коде (`682446275, 770511678, 778333031`), `/start` лишь
разблокирует канал «бот → пользователь». Любой другой chat_id игнорируется молча.

## Шаг 3. Создать Web Service на Render

**Важно:** именно **Web Service**, не Background Worker — на free-плане воркер не
живёт 24/7, а web-сервис с пингами на `/health` не засыпает.

Вручную:
1. Render → **New + → Web Service** → подключить репозиторий.
2. Настройки:
   - **Runtime:** Python 3
   - **Build Command:** `pip install -r requirements.txt`
   - **Start Command:** `python pirateswap_bot.py`
   - **Instance Type:** Free
3. **Environment → Add Environment Variable:**
   - Key: `PIRATE_TG_TOKEN`, Value: *новый токен от BotFather*
   - (порт `PORT` Render проставляет сам — бот его читает.)
4. **Create Web Service.**

Либо через `render.yaml`: **New + → Blueprint**, токен ввести в дашборде
(в blueprint он `sync: false` — в код не попадает).

---

## Шаг 4. UptimeRobot — чтобы Render не засыпал

Free Web Service засыпает после ~15 мин без HTTP-трафика. Пинг не даёт уснуть:

1. На Render скопируй URL сервиса, напр. `https://pirateswap-bot.onrender.com`.
2. UptimeRobot → **Add New Monitor**:
   - Type: **HTTP(s)**
   - URL: `https://<твой-сервис>.onrender.com/health`
   - Interval: **5 минут**
3. Save. `/health` отвечает `200 OK`.

## Что смотреть в логах Render (Logs)

- `HTTP-сервер слушает 0.0.0.0:<port>` — web-часть поднялась.
- `Поток-поллер запущен` + `Тихий стартовый скан` — мониторинг пошёл.
- `[Prisma Case] baseline: N шт.` где **N > 0** — бот реально видит сток
  (Prisma всегда в наличии). Если 0 — фильтр/сеть сломаны.
- `🔥 ДРОП` — сработал алерт.
- `429` / `HTTP 403/503 (Cloudflare?)` — притормаживаем (backoff отрабатывает сам).
- `⚠️ отброшено чужих предметов` — анти-ложная защита поймала сбой фильтра
  (алерт по ним НЕ ушёл).

Проверить, что бот жив, можно в любой момент: написать ему **/status**.

## Поведение, о котором стоит помнить

- **Эфемерная ФС Render.** При каждом рестарте/редеплое `notify_state.json`
  обнуляется → все трое снова `notify=true`. Кто хочет тишины — жмёт 🔕 заново.
  Для троих это сознательный компромисс (так и решили). `events.jsonl` тоже
  обнуляется при рестарте — если нужна история расписания надолго, периодически
  скачивай его из Render Shell.
- **Тихий старт.** На старте бот фиксирует baseline без алертов — иначе каждый
  редеплой слал бы «уже в наличии». Алерты идут только на дропы ПОСЛЕ запуска.
- **Cloudflare с серверного IP.** Темп оставлен спокойный (3–6 с между
  предметами, 30–50 с между обходами) + человеческий User-Agent + backoff на
  429/5xx. Если Render начнёт ловить `1015` — увеличь `ITEM_GAP_*` и `POLL_*`
  в начале `pirateswap_bot.py`.
- **Heartbeat** (`❤️ Бот жив`, раз в 6 ч) шлётся только админу (`770511678`).
- **Админ-алерт на серверные ошибки.** При HTTP 5xx или Cloudflare 403/503 админу
  (`770511678`) приходит ЛС `🛑 Серверная ошибка PirateSwap`. Чтобы при череде
  ошибок не было спама — не чаще одного сообщения раз в 15 мин
  (`ERROR_ALERT_COOLDOWN_MIN`).
